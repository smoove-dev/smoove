smoove has a small surface. Three ideas carry almost everything: the
composition is a frame clock, a sequence is a range-gated layer, and every tick
runs your updaters and repaints.

## Composition: the frame clock

A `Composition` _is_ a `Konva.Stage` with a clock bolted on. `fps` and
`durationInFrames` define a fixed timeline; the composition tracks a current
frame and advances it during playback.

```ts
const fps = 30;
const comp = new Composition({
  id: "scene", // unique id
  fps,
  durationInFrames: fps * 5, // frames 0…149
  width: 1280,
  height: 720,
  loop: true, // restart at 0 when the end is reached
});
```

Read the clock reactively at any time:

```ts
comp.frame.get(); // current frame
comp.isPlaying.get(); // true while playing
comp.durationInFrames.get(); // total length, same value you passed in above
comp.frame.subscribe((f) => console.log("now at", f));
```

`comp.props` is a reactive signal too: the scene's input data, read in updaters
with `comp.props.get()` and replaced at runtime with `comp.setProps(...)`. It's
how you parametrize one composition to render many videos. See
[Dynamic props](/docs/dynamic-props).

## Sequence: a range-gated layer

A `Sequence` _is_ a `Konva.Layer` pinned to a window on the timeline:
`[from, from + durationInFrames)`. It's only visible, and only repainted, while
the playhead is inside that window. Outside it, the layer is hidden and its
updaters don't run.

```ts
const intro = new Sequence({ from: 0, durationInFrames: 60 }); // frames 0…59
const outro = new Sequence({ from: 90, durationInFrames: 60 }); // frames 90…149
comp.add(intro, outro);
```

Both options have defaults. `from` is `0`, and `durationInFrames` is the
composition's own duration. So `new Sequence()` is a layer that spans the
whole timeline, which is all you need for a single-scene composition:

```ts
const scene = new Sequence(); // frames 0 … comp.durationInFrames - 1
```

This is how you compose a timeline out of independent pieces: stagger their
`from` values, overlap them, or lay them back-to-back. Each only does work while
it's on screen.

### Clip: a sequence you can nest

A `Sequence` is a layer, so it lives at the top of the composition. When you
want the same range-gating *inside* a scene, use a [`Clip`](/docs/clips). It
takes the same options, gates the same way, and nests to any depth, counting
frames local to whatever timeline holds it.

```ts
const card = new Clip({ from: 20, durationInFrames: 60 });
card.register((frame) => {
  // 0 on frame 20 of the sequence that holds this clip
});
scene.add(card);
```

### Local frame vs global frame

The updater you pass to `seq.register(...)` receives the **sequence-local**
frame. It's `0` on the sequence's first visible frame, counting up from there.
For a full-duration sequence (`from: 0`) the local frame equals the
composition's global frame; for a shifted sequence it's offset by `from`.

```ts
const outro = new Sequence({ from: 90, durationInFrames: 60 });
outro.register((local) => {
  // When the global frame is 90, `local` is 0.
  card.opacity(interpolate(local, [0, 30], [0, 1], { extrapolateRight: "clamp" }));
});
```

Local frames let you write a sequence once and place it anywhere on the timeline
without rewriting its ranges.

An updater gets a second argument with the rest of the frame's context, on both
`Sequence` and `Clip`:

```ts
outro.register((local, info) => {
  info.time; // local seconds: local / fps
  info.fps; // the composition's fps
  info.durationInFrames; // this timeline's own span
  info.globalFrame; // the composition frame
});
```

Reading `info.time` instead of the frame keeps motion at the same speed if you
change the composition's `fps` later.

## The tick loop

On every tick (during playback, or on a seek) the composition processes each
sequence that's in range, in this order:

1. **Updaters** run: your `register` callbacks set attributes from the frame
   (and from `comp.props.get()`, if the scene is parametrized).
2. **Nested timelines and media tick**: any [`Clip`](/docs/clips) in range runs
   its own updaters, and `Video`/`Audio` nodes sync to the frame.
3. **Layout**: `Flex`/`Block` roots recompute, so animated sizes/gaps reflow.
4. **Draw**: a `draw()` on the frame a sequence becomes active, `batchDraw()`
   after.

You only ever write step 1. The rest is the engine.

## Finding nodes

Keeping a reference to every node you animate is the usual way to work, and it
is the clearest one. When a node is out of reach, because it lives in another
sequence or inside a component you did not build, look it up by selector
instead. `Composition`, `Sequence`, and `Clip` all have `query` and `queryOne`:

```ts
comp.queryOne<Audio>("#music"); // by id, first match or null
scene.query(".bar"); // by name, every match in this subtree
```

Selectors are Konva's: `#id`, `.name`, or a type name like `Circle`. A query on
the composition searches the whole stage, and a query on a sequence or clip
searches its own subtree.

Results are cached until the scene graph changes, so calling one every frame
from an updater is fine. Here a voice-over sequence ducks a music track that
lives in a different sequence:

```ts
voiceOver.register((frame, info) => {
  const music = comp.queryOne<Audio>("#music");
  music?.volume(
    interpolate(frame, [0, 12, info.durationInFrames - 12, info.durationInFrames], [1, 0.25, 0.25, 1]),
  );
});
```

Set `id` and `name` when you construct a node. Renaming a node that is already
in the scene does not refresh the cache.

<Callout type="warn" title="One writer per property">
Two updaters that both write `music.volume()` will fight, and which one wins
depends on the order they were registered. Give every animated property a
single owner.
</Callout>

Nodes can also look upward, the way Konva nodes reach their stage with
`getStage()`:

```ts
node.getComposition(); // the owning Composition, or null while detached
node.getSequence(); // the sequence this node is in
node.getClip(); // the nearest clip, or null
```

These are how a component reads the context it was dropped into, `fps` and
duration being the common ones.

## `play()` in the browser, `setFrame(n)` anywhere

`comp.play()` drives the clock with `requestAnimationFrame`, so it's browser only
and throws in Node. `comp.setFrame(n)` jumps the playhead to a specific frame and
re-renders; it works in any environment, which is exactly what scrubbing and the
headless renderer rely on.

```ts
comp.play(); // browser playback
comp.pause(); // freeze at the current frame
comp.stop(); // pause and reset to frame 0
comp.setFrame(42); // seek anywhere, browser or Node
```

<Callout type="tip" title="Determinism is the payoff">
Because a frame's output depends only on the frame number (not on wall-clock
time or how you got there), `setFrame(42)` always paints the same thing, whether
you scrubbed to it, played to it, or rendered it on a server. Keep your updaters
pure and that guarantee holds. See [Best practices](/docs/best-practices).
</Callout>
