These are the recommended-way principles. Most have a working escape hatch.
They're here because following them keeps your animations deterministic,
seekable, and ready to render. Later pages link back to the relevant one where it
bites.

## Prefer the core wrappers over raw `Konva.*`

`Rect`, `Circle`, `Text`, `Image`, and the rest from `@smoove/core` extend
their Konva equivalents and add full flex participation: percentage sizes
(`width: "100%"`), flex child props (`flexGrow`/`flexShrink`/`flexBasis`/
`alignSelf`), and the richer nodes (`Text` fit and highlights, `Image`
`objectFit`, `Block` backgrounds). A raw `Konva.Rect` with a fixed pixel size is
still measured and placed inside a `Flex`, but it can't express any of those, so
it can't grow to fill a row, take a percentage width, or align itself. In the
demo below the container grows: the `flexGrow` wrapper fills it, the raw box
can't and leaves a gap.

<Demo name="wrapper-vs-raw" label="grow the container: the flexGrow wrapper fills it, the raw box can't" />

**Why:** the wrappers are the drawing vocabulary, and importing them gives every
node `%` sizes, flex child props, and the rich `Text`/`Image`/`Block` features.
Reach for `Konva.*` only for something core doesn't wrap yet.

```ts
// ✅ do: layout-aware
import { Rect } from "@smoove/core";
row.add(new Rect({ width: 120, height: 120, fill: "#4ea1ff" }));

// 🚫 avoid (unless intentional): renders, but never reflows in a Flex
import Konva from "konva";
row.add(new Konva.Rect({ width: 120, height: 120, fill: "#4ea1ff" }));
```

## Give the main sequence no options

`new Sequence()` for the sequence that spans the whole composition, no
arguments at all. `from` defaults to `0` and `durationInFrames` defaults to
the composition's own duration, so a full-span sequence needs nothing else.
Pass `from` and `durationInFrames` only when you want an actual partial
window.

**Why:** typing out the full range as explicit options gives you two numbers
that have to stay in sync with the composition's own duration by hand. Leaving
them unset means there's nothing to drift.

```ts
// ✅ do: the whole timeline, no options
const main = new Sequence();

// 🚫 avoid (unless intentional): a number that has to be kept in sync by hand
const main = new Sequence({ from: 0, durationInFrames: 90 });
```

## Size duration from fps, read it back from the composition

Write `durationInFrames` as `fps * seconds` at authoring time, not a bare
frame count, so the intent reads at the call site. At runtime, read the total
back with `comp.durationInFrames.get()` rather than hardcoding the same
number again somewhere else.

**Why:** `fps * 3` says "3 seconds" the moment you read it; `90` doesn't.
`durationInFrames` is a reactive signal, so `.get()` always reflects the
composition's actual duration, even if it's changed since the module loaded.

```ts
// ✅ do
const comp = new Composition({ id: "scene", fps, durationInFrames: fps * 3, width, height });

main.register((frame) => {
  const total = comp.durationInFrames.get();
  circle.x(interpolate(frame, [0, total], [0, width]));
});

// 🚫 avoid: two places that both have to agree on "90"
const comp = new Composition({ id: "scene", fps: 30, durationInFrames: 90, width, height });
main.register((frame) => {
  circle.x(interpolate(frame, [0, 90], [0, width]));
});
```

## Animate as a function of frame

Drive every attribute from the frame inside `seq.register()`, using
[`interpolate`](/docs/interpolation) / [`Easing`](/docs/easing). Never reach for
`setInterval`, `requestAnimationFrame`, `Konva.Tween`, or CSS transitions.

**Why:** only frame-derived values render deterministically and seek/scrub
correctly. A timer-driven value can't be reproduced at an arbitrary frame.

```ts
// ✅ do
seq.register((frame) => {
  box.x(interpolate(frame, [0, 30], [0, 400], { extrapolateRight: "clamp" }));
});

// 🚫 don't
setInterval(() => box.x(box.x() + 4), 16); // invisible to scrubbing and rendering
```

## Keep compositions pure

A composition should be a pure function of `frame` (and any `props`): no
external mutable state, no `Date.now()`, no random values that change between
renders.

**Why:** purity is what makes preview, scrubbing, and headless render produce
identical output. If you need randomness, seed it from the frame or a fixed seed.

External data belongs in [`props`](/docs/dynamic-props): read `comp.props.get()`
inside the updater and change it with `comp.setProps(...)`, which re-renders for
you. Pass a new object rather than mutating the old one; `setProps` skips the
re-render when the reference is unchanged.

## Add a node before you animate it

Attach a node to a `Sequence`/layer (`seq.add(node)`) before you set attributes
on it. Tweening a detached node updates its attrs but never repaints.

**Why:** the sequence is what draws the node. No parent layer, no paint.

## Pin one copy of `konva`

`konva` is a peer dependency and core extends its classes. Two copies in the
dependency tree break `instanceof` checks and layout.

**Why:** a node from copy A won't be recognized by code holding copy B. Dedupe in
the consumer (e.g. pnpm's single version, or a bundler `dedupe`).

## Let layout reflow; don't hand-position structure

Use `Flex`/`Block` and animate `gap`, size, or `padding`; the engine recomputes
layout each tick so children follow. Save manual `x`/`y` math for one-off motion,
not for structure.

**Why:** hand-positioned layout drifts out of sync the moment content or sizes
change. Layout that reflows stays correct for free.

```ts
// ✅ do: animate the container, children follow
row.setAttrs({ gap: interpolate(frame, [0, 30], [8, 64]) });

// 🚫 avoid: re-deriving each child's x by hand
children.forEach((c, i) => c.x(i * (120 + gap)));
```

## Position and size nodes relative to each other

When one node should track another, read the other node's own getters
instead of a guessed number: `next.x(prev.x() + gap)` keeps a fixed gap no
matter how `prev` moves. When two nodes need to match, read the computed box
with `getClientRect()` instead of copying a size by hand: it reports the
node's actual rendered bounds, so it stays correct as the source node
animates.

**Why:** this is the same "let it reflow" idea as layout, applied to plain
nodes outside a `Flex`/`Block`. A relationship expressed as a getter call
survives edits to either node; a relationship expressed as a hardcoded number
only survives until someone changes one side of it.

```ts
// ✅ do: name follows the avatar with a fixed gap, whatever the avatar's radius
name.x(avatar.x() + avatar.radius() * 2 + 12);
name.y(avatar.y());

// ✅ do: underline matches the title's rendered width, even as the title animates
underline.width(title.getClientRect().width);

// 🚫 avoid: numbers that only happen to be correct today
name.x(76);
underline.width(240);
```

## `play()` is browser-only; `setFrame(n)` works anywhere

Use `comp.setFrame(n)` for offline and server paths. It's also what the
headless renderer drives. `comp.play()` needs `requestAnimationFrame` and throws
outside the browser.

## Use `interpolate` extrapolation, not manual clamping

Reach for `extrapolateLeft` / `extrapolateRight: "clamp"` instead of wrapping a
ratio in `Math.min` / `Math.max`.

**Why:** the intent ("hold at the ends") reads clearly and stays at the call
site, and the easing is applied correctly within the range.

```ts
// ✅ do
interpolate(frame, [0, 30], [0, 1], { extrapolateRight: "clamp" });

// 🚫 don't
Math.min(1, frame / 30);
```

## Fake glows with gradients, not `shadowBlur`

In an animated scene, every active layer repaints on every frame, and canvas
`shadowBlur` re-runs a Gaussian blur for each shadowed shape each time. For a
glow effect, use a radial-gradient fill sized to the glow extent instead:
solid core, a mid-stop, then a transparent rim. Give a glowing line a second,
wider stroke at low opacity underneath the crisp one, and give a sharp shape
(a diamond, a star) a gradient disc behind it.

**Why:** in a glow-heavy scene we profiled, shadow blur cost more per frame
than everything else combined, and it gets worse with screen size and pixel
ratio. Gradient fills read the same and draw at a fraction of the price.
`shadowBlur` is fine for a scene that draws once and holds still.

```ts
// ✅ do: glow as a radial gradient (the mid-stop mimics a blur's falloff)
const spark = new Circle({
  x, y,
  radius: r * 3.5, // the glow extent, not the body
  fillRadialGradientStartPoint: { x: 0, y: 0 },
  fillRadialGradientEndPoint: { x: 0, y: 0 },
  fillRadialGradientStartRadius: 0,
  fillRadialGradientEndRadius: r * 3.5,
  fillRadialGradientColorStops: [0, color, 0.28, color, 0.5, `${color}40`, 1, `${color}00`],
});

// 🚫 avoid in animated scenes: one Gaussian blur per shape, per frame
const slow = new Circle({ x, y, radius: r, fill: color, shadowColor: color, shadowBlur: 10 });
```

One gotcha: gradient radii live in the node's local coordinates, so they
don't follow an animated `radius()`. To pulse or breathe a gradient shape,
animate `scale` and the gradient scales with the body.

```ts
// ✅ do: the gradient breathes with the body
orb.scaleX(k);
orb.scaleY(k);

// 🚫 avoid: the body grows, the gradient stays put
orb.radius(base + delta);
```

## Match `fps` to the motion

Pick 30fps for ambient motion (backdrops, slow loops, drifting particles) and
save 60fps for fast motion the eye tracks. And when a composition plays as a
page backdrop, cap the player's resolution with
[`max-pixel-ratio`](/docs/player#attributes).

**Why:** `fps` is the redraw rate, so it multiplies the cost of everything
above: half the frames, half the paint work. Slow motion reads identically at
30fps.
